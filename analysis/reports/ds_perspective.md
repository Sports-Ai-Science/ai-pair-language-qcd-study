# データサイエンティストの観点での再分析

**設計**: 2 群 × N=3、並列 sub-agent、Opus、独立コンテキスト
**統計手法**: Welch's t / Mann-Whitney U / Cliff's δ / Cohen's d / bootstrap CI / 検定力分析
**多重比較補正**: Bonferroni α = 0.05/10 = **0.005**

## 2. 重要な統計的観察

### 観察 1: 「bugs」の完全分離は統計的に異様
- EN 群 [1, 1, 1] vs JP 群 [0, 0, 0]
- 言語内分散が完全にゼロ（標準偏差 = 0）
- Mann-Whitney U で p=0.047（境界線）
- **N=3 で完全分離は非常に珍しい** — 偶然なら確率約 5%
- **可能性**: 真の効果 OR 「バグ」の定義が言語で違う（自己報告バイアス）

### 観察 2: 効果量は大きいのに p 値が緩い
- Cohen d > 1.0 が 8 指標
- これは「**効果は明白だが、N=3 では検出力が足りない**」典型例
- 必要な N: ほとんどの指標で **N=4〜12** で α=0.05 達成可能

### 観察 3: 言語内分散が言語効果の半分〜同程度
- Pooled SD (EN/JP 平均): active_min で 1.13 vs 言語効果 1.62 → **SNR=1.43**
- これは「言語選択 1 個分のノイズ ≈ Claude 生成自体のばらつき 1 個分」
- → **同じ言語でも、別 Claude インスタンスは違うものを作る**ことの証拠

## 3. 成果物を見て分かったこと（重要）

### 観察 4: アルゴリズム選択そのものが違う
```
calc-en-1: recursive-descent parser (再帰下降構文解析)
calc-jp-1: shunting-yard (操車場アルゴリズム)
```

→ **「言語効果」と思っていた差の一部は「アルゴリズム選択効果」かもしれない**。同じ AC でも、Claude が選ぶ実装手法が違う。これが LOC や js_lines の差に影響している可能性がある。

→ **真の言語効果を測るには、アーキテクチャを固定する必要があった**（pre-specify "use shunting-yard")。

### 観察 5: Test 名で言語が反映されている
```
EN: test('AC-01 addition', ...)
JP: test('AC-01 四則演算: 加算', ...)
```
- JP の test 名はより記述的（「加算」より「四則演算: 加算」）
- これが「JP の方がテスト数多い」の一因かも（同じ概念を細分化して命名）

### 観察 6: アサーション数の比較（より公平な指標）
| | EN | JP |
| --- | ---: | ---: |
| 総テスト数 | 128 | 160 |
| 総アサーション数 | 252 | 300 |
| アサーション/テスト | 1.97 | 1.88 |

→ **JP は「テスト数が多い」のではなく「同じことをより細かく分割している」**。実際の検証量（assertion 数）は 19% 多いだけで、テスト数 25% 増の主因はネーミングの粒度。

### 観察 7: ドキュメントは UTF-8 バイト数で誤解を生む
- JP docs: 29,364 bytes
- EN docs: 22,181 bytes
- 「JP docs が 32% 大きい」と思いがちだが
- **JP 文字は UTF-8 で 3 バイト/文字、EN は 1 バイト/文字**
- 文字数で比較すると JP docs は **EN の約 45%（半分以下）**
- → **JP は同じ情報を半分の文字数で書いている = 情報密度が高い**

## 4. 真の信号 vs 交絡因子

| 観測差 | 真の言語効果? | 交絡因子の可能性 |
| --- | --- | --- |
| JP の方が遅い (+1.6 min/run) | おそらく Yes | テスト数の多さ・コメント密度の高さ・ドキュメント密度の高さの累積 |
| JP の方がトークン多い (+35K) | Yes | UTF-8 トークナイザ + テスト多 + ドキュメント多 |
| JP の方が JS コード少ない (-85 行) | おそらく Yes | アルゴリズム選択の違いも含む（交絡） |
| JP の方がコメント多い (+8 行) | Yes | 文化的コーディングスタイル |
| JP の方がバグ少 (1→0) | **不明** | 「バグ」の定義が言語で違う可能性 |
| JP の方がドキュメント多い (バイト) | **誤読** | バイトでなく文字数で見ると JP は半分 |
| JP の方がテスト多い (+10 件) | 部分的に Yes | テスト名を細分化したナイーブな増加 |

## 6. データサイエンティスト的「気をつけろ」リスト

### 🚨 危険な解釈 1: 「JP の方が品質が高い」
- バグ 0/3 vs 1/3 は **統計的に偶然と区別不能**（MW p=0.047 は境界、Bonferroni 不通過）
- バグの自己定義差を考慮すると、結論として「JP 優位」は弱い
- **本当に欲しいのは: 第三者が両方の calculator を fuzz testing して escape rate を測ること**

### 🚨 危険な解釈 2: 「JP の方がコードが短い = 良い」
- 短いことが良いかは別問題。可読性・テスト容易性のトレードオフ
- アルゴリズム選択の交絡があるため、純粋に「言語が短くした」とは言えない

### 🚨 危険な解釈 3: 「JP の方がドキュメント多い（バイトで）」
- UTF-8 のバイト数で測ると JP は不利な見え方（情報量より符号化コスト）
- **文字数 or トークン数で測るべき**

### 🚨 危険な解釈 4: 「Cost で英語が 1.43x 安い」(前回主張)
- これは **total_tokens** の比。input が両者ほぼ同じなので、output の比はもっと大きい
- 実際の output 比は推定 ~2x
- **でも一方で「ユーザーが体感するコスト」は input+output 込みなので 1.43x が正しい**
- 視点によって正しい数字が違う

## 7. 整理された結論

### 確実に言えること（CI が 0 を除外）
1. **active_min**: JP は EN より **平均 2.6 分長い** [CI: +1.4, +3.8]
2. **js_lines**: JP は EN より **平均 85 行少ない** [CI: -137, -31]
3. **tokens_total**: JP は EN より **平均 35,000 多い** [CI: +13K, +58K]
4. **comment_lines**: JP は EN より **平均 8 行多い** [CI: +2, +15]
5. **src_loc**: JP は EN より **平均 115 行少ない** [CI: -214, -17]
6. **phase_sum_min**: JP は EN より **平均 1.6 分長い** [CI: +0.1, +3.2]

### 確証されていないこと
- バグ件数の差（境界 p、定義依存）
- テスト数の差（CI が 0 を含む）
- ドキュメント量（バイトで見るな、文字で見ろ）

### 完全に未検証
- 真の品質（escape defect rate、可読性、保守性）
- ユーザー（人間）の読解速度
- 長期保守コスト

## 8. 推奨される次のアクション（DS 視点）

### 短期: N を増やす
- N=10/group（合計 20 agent）で大半の指標を significant に持ち上げる
- 追加コスト ~$30, 約 30 分

### 中期: 交絡因子を統制する
- アルゴリズム選択を固定（"use Shunting-yard" を仕様に入れる）
- テスト数を仕様で固定（"write exactly 50 tests covering each AC"）
- これにより「言語の純粋効果」がより明確に

### 長期: 真の品質指標を導入
- Mutation testing で escape rate 測定
- 第三者（別 Claude インスタンス）に両 calculator をレビューさせ、可読性スコア化
- ユーザー（人間）に両ドキュメントを読ませて comprehension test

---

## 9. 修正版の最終結論（DS の言葉で）

> **「言語選択は QCD の Cost と Time に明白な効果（large effect, d > 1.5）を持つが、Quality 指標（バグ、テスト効果性、保守性）への効果は N=3 では結論不能。観測された差の一部はアルゴリズム選択や命名粒度の交絡を含み、純粋な言語効果ではない可能性がある。**
>
> **運用判断としては『Cost 重視なら English、丁寧さ重視なら Japanese』だが、両者ともに produce する成果物は機能的には等価（10/10 AC, 100% test pass）。最大の発見は『AI 生成は同じ仕様でも非決定的で、within-group 分散が between-group 効果と同程度になる場合がある』という方法論的事実。」**
